# ZETS current state (verified, not hype)

- Rust binary, 211,650 atoms + 13.2M edges from 331 Wikipedia articles. 158MB snapshot on disk.
- Graph-based: atoms = concepts; edges = typed relations (is_a, has_attribute, co_occurs_with, etc.) with provenance tags: Asserted | Observed | Learned | Hypothesis.
- Spreading activation + smart_walk. Deterministic, cross-process (10/10 identical runs verified).
- Benchmark: 32-question multi-domain MC test.
  - 236-atom baseline: 90.6%
  - 882-atom medium: 93.8%
  - 211,650-atom Wikipedia: 68.8% (wiki wins in physics, geography; struggles in chemistry, astronomy, biology — corpus coverage issue, not architecture)
- Direct-edge scoring: when a choice word's atom has many direct edges to seed atoms, it wins. This is the primary signal that made Wikipedia-scale viable.
- 5 moats measured: 100% determinism, ×6338 speed vs LLMs, 100% refusal on out-of-domain, 0 catastrophic forgetting, 100% audit trace per answer.
- NO LLM in the loop. NO neural nets. Deterministic, auditable, reproducible-from-git.

# Constraints

- Must run on: mobile phone (4GB RAM), laptop (16GB), server (64GB), edge chips.
- Must be offline-capable.
- Must stay deterministic.
- Must scale eventually to Wikipedia + PubMed + arXiv + Wiktionary + multi-language + culture/entertainment corpora.

## Q1 — Language model.

I want English + Hebrew as "core" (always loaded, uncompressed). Other languages as compressed packs with grammar-rules pre-loaded but vocabulary frozen. Unfreeze a language only when it starts getting real use.

- What defines "pre-load grammar-rules" concretely? Morphology tables? POS tagger data? Script+alphabet recognizers?
- How do I detect "start getting real use"? Token-frequency threshold? Query-hit count?
- What's the unfreezing cost profile?

## Q2 — Knowledge packs (compressed domain data).

Each domain (medicine / CS / math / geography / culture / entertainment / slang / law) as its own pack. Compressed on disk. Decompressed lazily only when a walk hits that cluster. Re-compressed after idle.

- Best compression scheme for graph data that maintains walk-accessibility?
- Do I compress whole pack, or per-cluster (10-100 atoms each)?
- What's "activation threshold" to decompress? What's "idle timeout" to re-compress?
- Critical: walks are <0.2ms. Zstd decompress of 1MB takes ~100ms. 500× slowdown if we decompress full pack per walk. How to avoid?

## Q3 — Personal graphs (multi-tenant).

One server must host many people — family (Shai + Ben + Or + Yam + parents), business (CHOOZ employees), individuals. Each has their own private graph PLUS shared global knowledge.

- What's the data model? Per-user overlay on global? Separate graph with shared atoms?
- ACL per edge? Per atom?
- How do I inherit from a family member's graph without leaking private data?
- Facebook/LinkedIn social-graph lessons applicable?

## Q4 — Edge / Cloud sync.

Edge device (phone) runs lean ZETS. Cloud runs heavy ZETS. Flow:
1. Edge queries local graph. Found → deterministic answer.
2. Not found → query cloud. Cloud has → return.
3. Cloud missing → cloud enters "learn this topic" mode (fetches Wikipedia/PubMed/etc), returns with "learning" flag and non-deterministic marker.
4. Edge caches result locally with provenance.

- What's the sync protocol? Delta? Full snapshot? Per-atom?
- How does cloud discover it needs to learn? (Count of "miss" queries on same topic?)
- Bandwidth constraints on phone — how small can sync deltas be?
- CRDTs applicable?

## Q5 — Walk-based lazy decompression.

A full zstd decompress is expensive. But if a walk only touches 20 atoms in a 10,000-atom pack, can we decompress only those 20? Seekable formats, chunked compression, etc.

- zstd-seekable real answer or hack?
- How small can a decompression-unit be before overhead dominates?
- mmap + compression: does any scheme exist that lets me mmap compressed data and read compressed pages on fault?
- Alternative: graph-level split — each "cluster" is its own tiny snapshot, load as needed.

## Q6 — Inference by analogy.

If user says "large land mammal", the system should infer 4 legs, tail, milk-producing, etc. from existing mammal knowledge, without needing that exact species.

- In a graph with is_a + has_attribute edges this should "just work" via walk. Is it really that simple?
- Risk: the inferred attributes should be tagged as Hypothesis, not Asserted, to avoid hallucination creep.
- Does this need its own module (e.g., "analogy_walk()") or is it a consequence of the existing smart_walk?
- Concrete example: given the graph has {dog is_a mammal, cat is_a mammal, mammal has_attribute four_legs}, if user asks about "capybara", graph should infer four_legs as Hypothesis.
