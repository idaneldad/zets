Q1:
  best: "Pre-load grammar-rules" for ZETS means loading the necessary linguistic resources for robust text processing: tokenization rules (including language-specific segmentation for non-whitespace-delimited scripts), morphological analyzers (lemmatizers/stemmers, inflection tables), and part-of-speech tagger models. These are essential for mapping diverse surface forms to canonical atoms and for understanding sentence structure. "Real use" should be detected by a combination of factors: a sustained token-frequency threshold for the new language's vocabulary appearing in queries, and a minimum number of successful `smart_walk` operations that specifically require atoms from that language pack. The unfreezing cost profile is primarily disk I/O and CPU for decompressing the language's vocabulary graph and loading it into ZETS's in-memory data structures, proportional to the vocabulary size.
  risk: The assumption that grammar rules can be fully isolated from vocabulary and that a frozen vocabulary can be effectively processed without its full grammatical context is overly simplistic for highly inflected or agglutinative languages.
  concrete: Universal Dependencies (UD) framework for cross-linguistic grammatical annotation.

Q2:
  best: For graph data requiring walk-accessibility, a compression scheme supporting *block-level random access* is critical. Zstd, configured with a custom dictionary trained on graph patterns and utilizing its block-level indexing capabilities, is a strong candidate. Compression *per-cluster* is the only viable approach for meeting the 0.2ms walk latency. A "cluster" should be a highly cohesive subgraph or strongly connected component, not an arbitrary grouping. The "activation threshold" for a cluster should be the first attempt to traverse an edge into or within that cluster. "Idle timeout" should be a configurable duration (e.g., 5-10 minutes) since the last access to any atom within the cluster, combined with system memory pressure heuristics to trigger re-compression.
  risk: The 0.2ms walk latency is extremely aggressive and will be consistently violated by any non-trivial decompression, even at the cluster level, unless the clusters are extremely small or pre-warmed.
  concrete: Zstd with custom dictionaries and `ZSTD_seekable` API (or manual block indexing).

Q3:
  best: The optimal data model is a *shared global graph* with *per-user overlay graphs*. Global atoms and edges reside in the shared graph, while private atoms and edges are stored in the user's overlay. Queries first check the user's overlay, then the global graph. ACLs should be *per-edge*, as an atom's privacy is determined by its connections. For example, "John Doe" (atom) might be public, but "John Doe has_email john@example.com" (edge) is private. Inheritance from a family member's graph without leaking private data can be achieved by creating *read-only, permissioned views* of specific subgraphs or by allowing traversal of specific edge types across user boundaries only with explicit consent and strict provenance tagging.
  risk: Managing fine-grained, edge-level ACLs across millions of edges will introduce significant overhead to graph traversal, potentially violating the 0.2ms latency requirement.
  concrete: Row-Level Security (RLS) concepts from relational databases, adapted for graph edge properties.

Q4:
  best: The sync protocol must be *delta-based* and operate at the *per-atom/per-edge* level. Changes should be tracked as a versioned log of additions, deletions, and property updates, efficiently serialized for transport. The cloud discovers it "needs to learn" by tracking a configurable threshold of "miss" queries for a specific atom or subgraph from multiple edge devices or its own internal queries. This prevents redundant learning and focuses resources. Bandwidth constraints on phones necessitate extremely small deltas, meaning only the minimal changed data is transmitted. CRDTs are highly applicable for managing concurrent updates to user-generated or annotated graph data, ensuring eventual consistency without complex server-side conflict resolution.
  risk: Introducing CRDTs and a "learning" mode in the cloud introduces potential for non-determinism and challenges in maintaining auditability if not meticulously designed to propagate provenance and resolve conflicts deterministically.
  concrete: Merkle trees for efficient delta detection combined with a CRDT library (e.g., `automerge-rs`) for managing concurrent graph modifications.

Q5:
  best: Zstd-seekable is a real answer, not a hack, but it requires careful implementation to manage seek tables and block boundaries. The smallest practical decompression unit, given the 0.2ms latency, is a *cluster* or *small subgraph* (e.g., 100-1000 atoms) that can be decompressed quickly. Decompressing individual atoms is too granular due to overhead. `mmap` with compression is not generally supported by operating systems for arbitrary compressed data; you would still need to decompress after a page fault. The most pragmatic alternative is a *graph-level split* where each "cluster" is stored as its own individually compressed file or block within a larger archive, loaded and decompressed on demand.
  risk: The overhead of managing thousands or millions of tiny compressed files or blocks, including file handles, metadata, and dictionary management, could easily negate the benefits of fine-grained decompression.
  concrete: Zstd's block-level compression and `ZSTD_seekable` API, combined with a custom index mapping graph clusters to byte ranges within a larger compressed file.

Q6:
  best: Yes, inference by analogy *should* "just work" via a well-configured `smart_walk` that understands inheritance. If "capybara is_a mammal" and "mammal has_attribute four_legs", a walk from "capybara" traversing `is_a` edges upwards and then `has_attribute` edges downwards will naturally discover "four_legs". Crucially, any inferred attributes *must* be tagged as `Hypothesis` with full provenance tracing the inference path (e.g., `Hypothesis(DerivedFrom(EdgeID_is_a, EdgeID_has_attribute))`). This does not require a separate module but rather a specific *strategy* or *mode* within the existing `smart_walk` algorithm that prioritizes inheritance-related edge types and correctly propagates provenance.
  risk: Over-inference or incorrect inference due to incomplete, ambiguous, or conflicting graph data, leading to a proliferation of inaccurate hypotheses that are difficult to manage or prune.
  concrete: A specialized `smart_walk` strategy that explicitly follows `is_a` (or similar inheritance) edges to propagate properties, ensuring all inferred edges are tagged `Hypothesis` and retain their derivation path.

cross_cutting_concerns:
  1.  **Latency vs. Granularity:** The aggressive 0.2ms walk latency is a pervasive constraint that directly conflicts with the overhead of lazy loading, decompression, and fine-grained security/multi-tenancy. This requires extreme optimization at every architectural layer.
  2.  **Provenance Propagation:** Maintaining and propagating detailed provenance tags (Asserted, Observed, Learned, Hypothesis, and their derivations) across all operations—including inference, learning, and multi-tenant sharing—is fundamental to ZETS's auditability and determinism claims and must be a first-class architectural concern.
  3.  **Memory Management & Eviction:** With dynamic loading of language and knowledge packs, a robust and intelligent memory management system (e.g., LRU/LFU with memory pressure awareness) is essential to stay within the strict RAM constraints of mobile and edge devices.

what_idan_is_missing:
  *   **Graph Schema and Evolution:** There's no mention of a formal graph schema. Without one, managing consistency, ensuring correct inference, and integrating diverse corpora will become an unmanageable mess. How are new atom types, edge types, and their properties defined and evolved?
  *   **Conflict Resolution for Learning:** When the cloud "learns" new information, how are potential conflicts with existing `Asserted` or `Observed` data handled? What is the deterministic strategy for resolving contradictions from different sources (e.g., Wikipedia vs. PubMed)?
  *   **Cold Start Problem:** The initial query to a new domain or language pack will always incur the full decompression and loading cost, potentially leading to a poor user experience. How is this initial latency managed or mitigated (e.g., pre-warming based on user profiles or common query patterns)?
  *   **Query Language/API:** Beyond "smart_walk," how do users or applications interact with ZETS to formulate complex queries, filter results, or specify walk constraints? The current description is very low-level.
